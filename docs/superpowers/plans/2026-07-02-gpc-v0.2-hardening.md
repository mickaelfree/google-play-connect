### Task 1: Read-path tests + visible discard failures

**Files:** Modify `internal/playapi/client.go`; add tests in `internal/cmd/edits_test.go`, `internal/cmd/listings_test.go`, `internal/cmd/bundles_test.go`; new `internal/playapi/warn_test.go`.

1. **Discard warnings:** in `internal/playapi`, add a package-level `var WarnWriter io.Writer = os.Stderr`. In `WithTransaction`'s error path and `WithReadOnlyEdit`'s deferred discard, replace `_ = c.DiscardEdit(...)` with: on error, `fmt.Fprintf(WarnWriter, "warning: failed to discard edit %s for %s: %v\n", editID, packageName, err)`. Committed behavior unchanged; the primary error still propagates.
   - Test `TestDiscardFailureWarns` (playapi): fake server where DELETE on the edit returns 500 and the mutation callback fails; point `WarnWriter` at a bytes.Buffer (restore via t.Cleanup); assert the buffer contains "failed to discard edit".
2. **Read-path command tests** (cmd package, straightforward happy paths against the fake server):
   - `TestEditsGet` — GET edit returns id+expiry; command output contains the id.
   - `TestListingsGet` — listing for one locale; output contains title.
   - `TestListingsDeleteRequiresConfirmAndDeletes` — without `--confirm` fails; with `--confirm`, DELETE hits the listings/{locale} path inside a committed transaction.
   - `TestBundlesList` — read-only edit (begin + discard asserted), output contains a version code.
3. `make check`; commit `test: read-path command coverage + warn on failed edit discard`.

### Task 2: `gpc metadata pull --prune`

**Files:** Modify `internal/cmd/metadata.go`; tests in `internal/cmd/metadata_test.go`.

1. New bool flag `--prune` on `pull` only, default false, help: "delete local listing locales and image manifests that no longer exist remotely (local image files are never touched)".
2. After a successful pull (inside the same RunE, after the WithReadOnlyEdit block), when `--prune` is set:
   - Remove each `metadata/<app>/listings/<locale>/` directory whose locale was NOT in the pulled listings.
   - Remove each `metadata/<app>/images/<locale>/<type>.manifest.json` that this pull did not (re)write.
   - NEVER remove image files or image directories (push material).
   - Report pruned paths in the command output (JSON key `pruned`).
3. Tests: `TestMetadataPullPruneRemovesStaleLocale` (pre-create a stale `listings/de-DE` + stale `images/en-US/icon.manifest.json`; pull with `--prune` from a server exposing only fr-FR listings and no icon images; assert stale dir+manifest gone, fresh files present, and a pre-existing local image file under `images/en-US/phoneScreenshots/1.png` untouched). `TestMetadataPullNoPruneKeepsStale` (default: stale locale dir survives).
4. `make check`; commit `feat: metadata pull --prune (stale listings/manifests, opt-in)`.

### Task 3: Completed-release retention on staged publishes

**Files:** Modify `internal/cmd/releases.go`; tests in `internal/cmd/tracks_test.go`.

**Design (controller decision, from final-review finding):** when `releases publish` creates a STAGED release (`--rollout` set → status inProgress), gpc now FETCHES the track inside the transaction and RETAINS existing releases with status `completed` alongside the new staged release (Google requires the previous full release to stay listed while a staged one serves a fraction). New flag `--no-retain` restores the old replace-everything behavior. Full-rollout publishes (no `--rollout`) keep replacing the whole list — a new completed release supersedes everything by definition.

1. In publish's RunE, when `status == inProgress && !noRetain`: inside WithTransaction, `GetTrack` first; keep `existing.Releases` entries with `Status == playapi.ReleaseStatusCompleted`; final `Releases` = retained + new release. When the fetch 404s (empty/new track), proceed with just the new release. Any other fetch error aborts.
2. Flag `--no-retain`, default false, help: "staged publishes: replace the track's whole release list instead of retaining the current completed release".
3. Tests:
   - `TestReleasesPublishStagedRetainsCompleted` — GET tracks/production returns one completed release (versionCodes 40) + one halted; PUT body must contain exactly 2 releases: the retained completed (40) and the new inProgress (42 @ 0.25); halted NOT retained.
   - `TestReleasesPublishStagedNoRetain` — same server, `--no-retain`: PUT body has only the new release.
   - `TestReleasesPublishFullReplacesList` — no `--rollout`: no GET issued (handler for GET t.Errorf's), PUT has only the new completed release.
4. Update the gpc-release-flow skill (`skillsfs/skills/gpc-release-flow/SKILL.md`) staged-rollout section and README's API-limitations bullet to describe retention-by-default + `--no-retain`.
5. `make check`; commit `feat: retain completed releases on staged publishes (--no-retain to opt out)`.
